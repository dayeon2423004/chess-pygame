# チェスゲーム (Python)

> 単なるアルゴリズム実装ではなく、  
> 設計 → 失敗 → リファクタリング → 構造改善まで経験したプロジェクト

---

## プロジェクト概要

Pythonでチェスゲームを実装した個人プロジェクトです。  
複雑なルールと状態管理を通して、システム全体の設計力を向上させることを目的に開発しました。

---

## 開発背景

- これまで単純なアルゴリズム問題は解いていたが  
  **全体設計や複雑な問題の扱い経験が不足していると感じた**
- 状態管理とルールが複雑なテーマとして  
  **チェスゲーム**を選定

---

## 初期の問題点（1次開発）

### 1. 設計不足

- 駒の状態を画像パスで管理
- サーバー側での状態管理オブジェクトが存在しない
- クライアント依存のデータ構造

👉 結果  
- 状態の不整合発生  
- コードの複雑化  

---

### 2. モジュール化・責務分離不足

- メッセージ処理とゲームロジックが混在
- ゲームループ内で直接メッセージ処理

👉 結果  
- モノリシック構造（単一ファイル集中）
- 変更が他機能へ影響

---

## プロジェクト中断理由

- 構造的な限界
- 保守性の低さ
- 拡張性の欠如

👉 一度開発を中断

---

## リファクタリング（2次開発）

### 改善方針

- 責務分離（Separation of Concerns）
- 低結合（Low Coupling）
- データ構造の明確化

---

### 1. Boardオブジェクト導入

```python
board = {
    (0,0): "rook",
    (0,1): "knight",
}
```

- サーバー中心の状態管理へ変更
- 画像ベース → 座標ベースに改善

👉 効果  
- データの一貫性向上  
- 状態管理の明確化  

---

### 2. モジュール分割

従来:
```bash
game.py（すべてのロジック）
```

改善後:
```bash
game/
 ├── board.py
 ├── piece.py
 ├── game_logic.py
 ├── message_handler.py
```

👉 効果  
- 機能ごとの独立性向上  
- 保守性改善  

---

## 振り返り

### 1. 設計の重要性

- 実装より設計がプロジェクト全体に大きく影響

👉 学び  
設計は「後でやるもの」ではなく「最初にやるべきもの」

---

### 2. 保守性と拡張性

- リファクタリング後は  
  機能追加前に構造を考える習慣が身についた

---

### 3. 型ヒントの重要性

```python
def move_piece(position: tuple[int, int]) -> None:
    ...
```

- 動的言語特有の型エラーを防止
- 可読性と安全性向上
